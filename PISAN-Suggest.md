## Project Summary

UW Compass is a Next.js 16 (App Router) + React 18 + Tailwind web app that helps UW students find official campus resources by typing their situation in plain language. The AI pipeline (in `lib/ai.ts`) uses OpenAI `text-embedding-3-small` for semantic retrieval plus `gpt-4o-mini` with typed function-calling for both need extraction and per-resource explanations, combined with a multi-signal ranker in `lib/recommend.ts` over 31 curated resources in `data/resources.json`. Two contributors: `Dhruva` (built the MVP, ranker, tests, project website, campus filter, ~7,175 LOC added) and `Aryan Khanna` (README, UI redesign, progress bar, ~421 LOC added).

### AI Integration (15 pts)

AI is genuinely central, not bolted on. Three OpenAI calls happen per request (`app/api/recommend/route.ts:44,58`) and they are integrated into the core retrieval logic, not used as a side-chat:

1. `embed()` in `lib/ai.ts:20-26` produces a 1536-dim query vector.
2. `extractNeeds()` in `lib/ai.ts:107-125` uses OpenAI function-calling with a strict JSON schema (`NEED_EXTRACTION_TOOL`, `lib/ai.ts:36-88`) to produce typed `ExtractedNeed[]` with category, intensity, evidence, tags, and urgency. Forcing `tool_choice` with a typed schema is the right pattern and avoids JSON-parsing fragility.
3. `summarize()` in `lib/ai.ts:181-226` produces per-resource explanations and a 2-4 step plan, again via typed function-calling.

The classical (non-LLM) ranker in `lib/recommend.ts:53-123` then combines normalized cosine similarity (0.5), category match (0.25), tag overlap (0.15), and an urgency boost (0.1), followed by category diversification. This is a sound hybrid retrieval design and goes well beyond a chat wrapper. Calls 1 and 2 run in parallel via `Promise.all` (`route.ts:44`), which shows attention to latency.

Minor caveats: only 31 resources means the "embedding" signal is doing less work than it would at scale; the system is closer to "structured tagging assisted by embeddings" than full RAG, but for the corpus size that is a defensible choice.

**Estimated score: 14/15**

## Suggested Improvements & New Features

### Code Quality & Technical

- **Deploy publicly and put the URL in the README** (`README.md:7`). The rubric explicitly weights public deployment; without it, both "Technical Execution" and "Project Web Presence" lose points. Vercel + the existing build works.
- **Add rate limiting to `app/api/recommend/route.ts`**. Even a simple in-memory token bucket keyed by IP (or `@upstash/ratelimit` with Vercel KV) would protect the OpenAI key from runaway costs on a public URL. Right now anyone who finds the deployed URL can burn through your quota.
- **Add a React error boundary** wrapping `app/page.tsx` so a thrown render error shows a friendly message instead of a white screen.
- **Handle the OpenAI 429/quota path explicitly** in `route.ts:80-84`. Currently it returns the raw error message to the client; a friendlier 503 with "service temporarily unavailable" is better UX.
- **Add a small response cache** keyed by `hash(input)+campus` (e.g. `Map` with TTL or Vercel KV). Many students will phrase identical things; caching saves both latency and OpenAI cost.
- **Empty `next.config.js`** — at minimum add `reactStrictMode: true`. Consider adding security headers (CSP, X-Frame-Options) via `headers()`.
- **CI**: add a `.github/workflows/ci.yml` that runs `npm run typecheck` and `npm test` on every PR. The tests exist; automating them prevents regressions.
- **`scripts/check-links.ts`** should run in CI on a schedule (weekly) so the team learns about UW URL changes without manual prompting; this directly addresses the "resources go stale" risk from the proposal.
- **The `Recommendation.matched_needs` field in `lib/recommend.ts:95`** only ever holds a single category. Either change the type to `Category | null` or actually collect all matching needs.
- **Document the `data/resources.embedded.json` regeneration policy** in the README a bit more loudly: someone editing `resources.json` and forgetting to run `npm run seed` will silently ship stale embeddings.

### UI / UX Improvements

- **Loading skeleton for the results panel** would feel more responsive than just the bottom progress bar (`app/page.tsx:136-140`). Render placeholder cards while the request is in flight.
- **Persist the last query and campus selection in `localStorage`** so a refresh doesn't lose work. This also enables the README's "Saved recommendations" stretch goal.
- **Mobile**: the campus pill row (`app/page.tsx:77-103`) wraps awkwardly on narrow viewports; consider a select dropdown below 480px.
- **Accessibility**: confirm color contrast for `text-uw-spirit-purple` on light backgrounds (the badge text in `app/page.tsx:205` is small and may fail WCAG AA). Add `aria-live="polite"` to the results section so screen readers announce when recommendations appear.
- **Add a "copy to clipboard" or "share" button** on each recommendation card. Students helping friends will want to send a link or a summary.
- **Show a per-need confidence indicator** (currently only `intensity` is shown). The model's tag list is hidden; surfacing it as small chips would help students understand why a resource matched.

### New Features

- **Feedback buttons** ("This was helpful" / "Not what I needed") — already listed as a stretch goal in `README.md:166`. Even without a backend, log to an append-only JSON file or Vercel KV; the data is gold for tuning the ranker weights.
- **"Compare campuses" view** for transfer / multi-campus students: same query, three columns of recommendations.
- **iCal export of next steps**: turn the AI-generated `next_steps` into calendar events ("This week: visit OWRC drop-in").
- **Anonymous query gallery** (opt-in): show recent example queries so students realize what kinds of things they can ask.
- **Multilingual input** (already a stretch goal): GPT-4o-mini handles this natively; you mostly need to translate the UI strings and confirm the tag vocabulary survives.
- **Resource-side "what I'm experiencing" reverse lookup**: a static page per resource that lists which situations route to it (good for SEO + a useful exercise for office staff).
- **A "for advisors" mode** with a richer output (full scores, all matched tags, alternative options) for academic-advising staff using this as a triage tool.

### AI Integration Enhancements

- **Two-pass extraction with self-critique**: after `extractNeeds`, do a cheap second pass asking the model "Did you miss anything? Is any need over-stated?". Will reduce false-urgent flags in particular.
- **Conversational follow-up**: after the first response, allow the student to refine ("I can't make evening appointments"). Currently the API is stateless; add an optional `prior_needs` field to `route.ts` and merge.
- **Function-calling for resource selection itself**: give the model a `recommend_resources(ids[])` tool with the full resource list and let it choose, instead of (or in addition to) the cosine-based ranker. Comparing the two would be a great experiment for the final report.
- **Persona-aware ranking**: add an optional `student_context` (year, campus, commuter, first-gen) field and pass it to need extraction; this maps directly to the existing tag vocabulary.
- **Eval harness**: extend `tests/scenarios.test.ts` with a labeled set of ~30 realistic student inputs and the resources you'd expect in the top 5. Run a nightly script that calls the real `extractNeeds` + ranker and reports recall@5; this gives you data-driven evidence for the final presentation.
- **Cite the matched span**: have `summarize()` quote 1-3 words from the student's input in each explanation (the system prompt asks for this, but it's not enforced). A second tool param `evidence_quote: string` would force the behavior.
- **Cost / latency logging** in `route.ts`: log `prompt_tokens`, `completion_tokens`, and total latency. Hard to optimize what you don't measure, and these numbers will look great in the final report.
